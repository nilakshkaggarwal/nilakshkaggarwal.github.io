    Data Security Rules

    Syntax

    using scope (<scope>) can be used to further refine data

    Defaults

    Name Default Access
    Dimension Read Only. Write Denied
    Attribute Read Only. Write Denied
    Plan No Access
    Measure No Access
    Model No Access

    Examples

    grant read access Rule_1 to roles [DataRole],[UIRole] for measure [UnitPrice];

grant write access Rule_2 to roles [DataRole],[UIRole] for plan Demand;

deny read access Rule_3 to roles [DataRole],[UIRole] for dimension [Customer];

grant write access Rule_4 to roles [DataRole],[UIRole] for plan Demand using
scope ([Location].[State].[Karnataka]);

    Note: After adding or editing any data security rule always execute the following query to reset and imply the changes made for the user workspace.

    resetaccesscontrol;

    Writing Rules

    From syntax it is clear that their are 3 ways to define rules

    1. deny read
    2. grant read
    3. grant write
    4. deny write doesn't exist as the defaults restrict access for all

    From Here on
    [Role_1]:UI Role [Role_2]:DATA Role

    Plan

    From default we know that we have no access. So, We have to grant access for plans

    Read Access

    Example:

    grant read access Access_to_SalesPlan to roles [Role2] for plan [Sales Plan];

    Result:

    Example:

    grant read access Access_to_SalesPlan_for_Product to roles [Role2] for plan 
[Sales Plan] using scope(Product.[ProductID].filter
(#.Name in {[FUR-BO-10002545], [FUR-BO-10004695]}));

    Result:

    Write Access

    User can edit the members of all the measures of all the measure groups that comes under a prticular plan after having write access for them

    Example:

    grant write access Plan_write to roles  [Role2] for Plan [Sales Plan];

    Result:

    Note: in above example measures NewPrice, Sold, Cost belongs to measure group "ItemGroup" and measure UnitPrice belongs to measure group "Product" and both the measure group comes under plan "Sales Plan".

    Model (Measure Group)

    Just like plan, model's need to be using grant read and grant write. Only difference is the access will be to a particular measureGroup.

    few examples: Read

    grant read access Access_to_ItemGroup to roles
 [Role2] for model [ItemGroup];

    Result:

    Note: Measure group "ItemGroup" contains only the measure related to the item report hence only data for item report is visible and not for UnitPrice report.

    Example:

    grant read access Access_to_Brand_notColgate to roles  [Role2] for model 
[Item] using scope(Item.[Brand].filter(#.Name != "Colgate"));

    Result:

    Note: Usage of scope to give access to all brands except 'Colgate'

    Similarly for write

    Example:

    grant write access model_write to roles  [Role2] for model [ItemGroup];

    Result:

    Note: in the above example user can edit the measures NewPrice, Cost, Sold as all three comes under the measuregroup "ItemGroup".

    Measure

    Similar to plan or model but gives access to one measure only

    Examples:read

    Example:

    grant read access Access_to_NewPrice to roles  
[Role2] for measure [NewPrice];

    Result:

    Example:

    grant read access Access_to_NewPrice_Brand_notColgate to roles  [Role2] 
for measure [NewPrice] using scope(Item.[Brand].filter(#. Name != "Colgate"));

    Result:

    Examples:Write

    In this case user can edit the members of a particular measure.

    Example:

    grant write access measure_write to roles  [Role2] for
 measure [NewPrice];

    Result:

    Note: in the above example user can edit only the members of measure NewPrice as access is given for only that measure.

    Dimension

    The default of dimension is having Read Access only. In this case one has to either grant write access or deny read access.

    Read Access Denial

    deny read access dimension_denied to roles  [Role2]
 for dimension [Product];

    Result:

    In the above workspace, there are more than one user and for some users we don't want to show the product data in above example. Hence we have denied the product dimension access.

    Write Access

    Giving write access to user allows the user to add or edit members to an attribute and data to the related measures of that dimension.

    Note: the changes in dimension member are made through action button which exists on the user workspace.

    Example:

    grant write access write_dimension_access to roles 
 [Role2] for dimension CNID;

    result:

    Sometimes, designer provide some action button on workspaces to add members. User need to add members in the particular dimension. So, designer can give user the write access to the dimension

    Note: in the above example the CNID is the attribute name where TK_6027 is a auto generated member which is added as the user submit the create Cn Token

    After the member is created user can add the required details of the measures through clicking on the add products action button.for the above example user can add values for measures LB invoice, Billed Qty etc.

    Attributes

    Similar to dimension attributes uses deny read and grant write to control access for a particular attribute of dimension. Other functions like filter can be used to filter some members of the attribute

    Example:Write grant write for attribute is similar to grant write for dimension but In this case the user is given the access to add member to a particular attribute and not for all the attribute of a dimension.

    Example:

    grant write access write_attribute_access to roles 
 [Role2] for levelattribute Product.[SubCategory];

    Result:

    Similar to the previous example user can add members to the attribute Subcategory under product dimension though the action button is available on user workspace.
