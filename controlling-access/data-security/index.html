    Data Security Original

    grant read

    For plan

    granting read access for a plan provides the user data viewing access for the whole plan which contains different measure group and measures under it.

    Example:

    grant read access Access_to_SalesPlan to roles 
 [Role2] for plan [Sales Plan];

    Result:

    Example:

    grant read access Access_to_SalesPlan_for_Product to roles  [Role2] for plan 
[Sales Plan] using scope(Product.[ProductID].filter
(#.Name in {[FUR-BO-10002545], [FUR-BO-10004695]}));

    Result:

    Note: User can give access to the selective members by using scope. But in scope only one dimension can be used

    For model (measure group)

    granting read access for model provides the data viewing access for the particular measure group along with all measures in it.

    grant read access Access_to_ItemGroup to roles
 [Role2] for model [ItemGroup];

    Result:

    Note: Measure group "ItemGroup" contains only the measure related to the item report hence only data for item report is visible and not for UnitPrice report.

    Example:

    grant read access Access_to_Brand_notColgate to roles  [Role2] for model 
[Item] using scope(Item.[Brand].filter(#.Name != "Colgate"));

    Result:

    Note: Usage of scope to give access to all brands except 'Colgate'

    For Measure

    granting read access for measaure provides the data viewing access for the particular measure data.

    Example:

    grant read access Access_to_NewPrice to roles  
[Role2] for measure [NewPrice];

    Result:

    Example:

    grant read access Access_to_NewPrice_Brand_notColgate to roles  [Role2] 
for measure [NewPrice] using scope(Item.[Brand].filter(#. Name != "Colgate"));

    Result:

    Deny read

    This case is implied usually when a workspace is shared with more than one users and one might need to restrict access of some data for some particular users.

    For Dimension

    deny read access dimension_denied to roles  [Role2]
 for dimension [Product];

    Result:

    Note: let's assume for the above workspace their are more than one user and for some users we don't want to show the product data in above example.hence we have denied the product dimension access.

    For members

    Example:

    deny read access attribute_denied to roles  [Role2] for members 
Item.[SKU].filter(#. Name =="Attraction");
deny read access member_denied to roles  [Role2] for members
 Product.[SubCategory].filter(#. Name =="Bed");

    Result:

    Note:

    Grant write

    For Dimension

    Giving write access to user allows the user to add or edit members to an attribute and data to the related measures of that dimension.

    Note: the changes in dimension member are made through action button which exists on the user workspace.

    Example:

    grant write access write_dimension_access to roles 
 [Role2] for dimension CNID;

    result:

    Sometimes, designer provide some action button on workspaces to add members. User need to add members in the particular dimension. So, designer can give user the write access to the dimension

    Note: in the above example the CNID is the attribute name where TK_6027 is a auto generated member which is added as the user submit the create Cn Token

    After the member is created user can add the required details of the measures through clicking on the add products action button.for the above example user can add values for measures LB invoice, Billed Qty etc.

    For attribute

    grant write for attribute is similar to grant write for dimension but In this case the user is given the access to add member to a particular attribute and not for all the attribute of a dimension.

    Example:

    grant write access write_attribute_access to roles 
 [Role2] for levelattribute Product.[SubCategory];

    Result:

    Similar to the previous example user can add members to the attribute Subcategory under product dimension. through the action button available on user workspace.

    For measure

    In this case user can edit the members of a particular measure.

    Example:

    grant write access measure_write to roles  [Role2] for
 measure [NewPrice];

    Result:

    Note: in the above example user can edit only the members of measure NewPrice as access is given for only that measure.

    For Model (measure group)

    In this case user can edit the members of all the measures that comes under a particular measure group.

    Example:

    grant write access model_write to roles  [Role2] for model [ItemGroup];

    Result:

    Note: in the above example user can edit the measures NewPrice, Cost, Sold as all three comes under the measuregroup "ItemGroup".

    For Plan

    In this case user can edit the members of all the measures of all the measure groups that comes under a prticular plan.

    Example:

    grant write access Plan_write to roles  [Role2] for Plan [Sales Plan];

    Result:

    Note: in above example measures NewPrice, Sold, Cost belongs to measure group "ItemGroup" and measure UnitPrice belongs to measure group "Product" and both the measure group comes under plan "Sales Plan".
